Modify the existing Kinross Offer Letter Companion web app.

This is a targeted additive correction.
Do not redesign the app.
Do not rewrite the scenario engine.
Do not paraphrase clause text.
Do not summarize source offer letters into “equivalent” language.
Do not normalize wording except through explicit token substitution.

==================================================
PRIMARY CORRECTION
==================================================

The app is currently paraphrasing source offer-letter language.
That behavior must stop.

From now on, the clause library must store and render the exact source language from the approved offer-letter templates.
The app may only do three things to source language:

1. store it exactly
2. replace variable placeholders with token placeholders
3. suppress clauses that are not applicable

No other rewriting is allowed.

==================================================
SOURCE OF TRUTH RULE
==================================================

The uploaded offer-letter files are the source of truth for clause text.

For each supported scenario, create clause records whose text is copied exactly from the relevant source template.
Do not “improve” wording.
Do not shorten.
Do not modernize punctuation.
Do not merge nearby clauses unless they are already one paragraph in the source.
Do not remove repeated phrasing if it appears in the source.

If the source template contains editorial comments such as “Round Mountain only” or “Comment by Chelly Koeppen”, store those comments separately as metadata flags or implementation notes.
Do not render those comments in the final offer letter unless the business explicitly wants them shown.
The displayed clause text must be the actual letter language only.

==================================================
CLAUSE STORAGE MODEL
==================================================

For every clause record in the clause library, store:

- clause_id
- scenario
- clause_number_source
- clause_title_internal
- exact_source_text
- tokenized_text
- source_file_name
- source_file_reference
- applicability_rule
- optional_flag
- source_comments_internal
- render_default
- sort_order

Rules:
- exact_source_text must be the verbatim paragraph from the source file
- tokenized_text must be the same exact text with only placeholders converted into app tokens
- tokenized_text must preserve all non-placeholder wording exactly
- source_comments_internal stores comments/instructions from the template but does not render to end users
- final rendering must come from tokenized_text, not paraphrase logic
